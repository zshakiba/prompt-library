# Project Framework

Version: 1.0

---

# Purpose

Projects are where learning becomes competence.

Every project should reinforce concepts, expose misconceptions, and build confidence.

Projects are not assignments to complete.

Projects are opportunities to think, experiment, fail safely, and improve.

---

# Core Principles

Every project should:

- Have a real-world purpose.
- Reinforce previous lessons.
- Introduce one new challenge.
- Encourage experimentation.
- Require decision making.
- Include debugging opportunities.
- End with reflection.

Avoid projects that only ask the learner to copy instructions.

---

# Project Progression

Projects increase in difficulty over time.

Level 1 — Guided

The learner follows detailed instructions.

Level 2 — Assisted

The learner makes small decisions while receiving guidance.

Level 3 — Independent

The learner solves most problems with occasional hints.

Level 4 — Open-ended

The learner designs and implements a solution.

Level 5 — Production

The learner builds a realistic project using industry practices.

---

# Standard Project Structure

Every project should contain the following sections.

## 1. Context

Describe a realistic scenario.

Explain why the project exists.

---

## 2. Goal

Define the expected outcome.

The goal should be measurable.

---

## 3. Learning Objectives

List the concepts the learner should practice.

Example:

- Docker Images
- Volumes
- Networking

---

## 4. Requirements

Describe functional requirements.

Example:

- Build an image
- Expose a port
- Persist data

---

## 5. Constraints

Add realistic limitations.

Examples:

- Maximum image size
- Limited memory
- Non-root user
- No internet access

Constraints encourage creative problem solving.

---

## 6. Success Criteria

Define what success looks like.

The learner should know when the project is complete.

---

## 7. Hints

Provide optional hints.

Never reveal the full solution immediately.

Hints should gradually increase in specificity.

---

## 8. Stretch Goals

Offer optional improvements for advanced learners.

These should extend—not replace—the core project.

---

## 9. Production Improvements

Explain how professionals would improve the project.

Examples:

- Security
- Performance
- Monitoring
- Logging
- Scalability

---

## 10. Reflection

Ask the learner questions such as:

- What was difficult?
- What surprised you?
- What would you change?
- What did you learn?
- How would this work in production?

Reflection is required.

---

# Debugging

Every project should contain at least one intentional problem.

Examples:

- Wrong port
- Permission error
- Missing dependency
- Incorrect environment variable

The learner should debug instead of receiving the answer.

---

# Assessment

Evaluate:

- Understanding
- Decision making
- Debugging ability
- Code quality
- Explanation quality

Do not evaluate memorization.

---

# Portfolio Quality

A project is portfolio-ready when it includes:

- Clear documentation
- Clean structure
- Meaningful commit history
- Error handling
- Production considerations
- Lessons learned

---

# Continuous Improvement

Every completed project should end with:

- Self-review
- Mentor feedback
- Refactoring
- Optional extension
- Connection to the next project

Projects are stepping stones, not finish lines.
